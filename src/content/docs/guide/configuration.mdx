All semrel behavior is controlled by a single config file in the root of your repository. You can specify a different path with `--config`.

## Supported formats

semrel auto-discovers the config file in this order:

| File | Format |
| --- | --- |
| `.semrel.yaml` / `.semrel.yml` | YAML |
| `.semrel.toml` | TOML |
| `.semrel.json` | JSON |

The examples on this page use YAML. TOML and JSON use `snake_case` keys (e.g. `tag_prefix` instead of `tagPrefix`).

## Full example

```yaml
version: 1
tagPrefix: v

branches:
  - name: main
  - name: "*.x"
    maintenance: true
  - name: next
    prerelease: beta

rules:
  - type: feat
    bump: minor
  - type: fix
    bump: patch
  - type: perf
    bump: patch
  - type: revert
    bump: patch

version_ceiling: "1.0.0"
ceiling_strategy: clamp

plugins:
  - uses: github
    name: github-release
    args:
      token: ${{ env.GITHUB_TOKEN }}
      owner: MyOrg
      repo: my-repo

  - uses: slack-notify
    path: /usr/local/bin/semrel-plugin-slack
    args:
      webhook_url: ${{ env.SLACK_WEBHOOK }}
```

## `version`

The config schema version. Always set this to `1`.

## `tagPrefix`

The prefix prepended to the version number when creating git tags.

```yaml
tagPrefix: v   # creates tags like v1.2.3 (default)
tagPrefix: ""  # creates tags like 1.2.3
```

| Value | Default | Description |
| --- | --- | --- |
| any string | `v` | Prepended to every tag semrel creates |

## `branches`

A list of branches from which releases can be made. Commits on unlisted branches are ignored.

```yaml
branches:
  - name: main
  - name: "*.x"
    maintenance: true
  - name: next
    prerelease: beta
```

### Branch fields

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `name` | string | — | Branch name or glob pattern (e.g. `main`, `"*.x"`, `release/*`) |
| `prerelease` | string | — | Pre-release identifier appended to the version (e.g. `beta` → `1.0.0-beta.1`) |
| `maintenance` | bool | `false` | Restricts this branch to patch-only bumps. Also auto-detected from `N.x` / `N.M.x` patterns |

<Aside type="tip">
  Maintenance branches like `1.x` or `2.3.x` are auto-detected — you don't need to set `maintenance: true` explicitly for name patterns that match `N.x` or `N.M.x`.
</Aside>

## `rules`

Maps conventional commit types to semver bump levels. If omitted, semrel uses the built-in defaults.

```yaml
rules:
  - type: feat
    bump: minor
  - type: fix
    bump: patch
  - type: perf
    bump: patch
  - type: revert
    bump: patch
```

### Rule fields

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `type` | string | yes | Conventional commit type (e.g. `feat`, `fix`, `perf`) |
| `bump` | string | yes | One of `major`, `minor`, `patch` |

<Aside type="note">
  Breaking changes (`BREAKING CHANGE` footer or `!` suffix like `feat!:`) always result in a `major` bump regardless of rules.
</Aside>

### Default rules

When `rules:` is absent, semrel uses:

| Type | Bump |
| --- | --- |
| `feat` | `minor` |
| `fix` | `patch` |
| `perf` | `patch` |
| `revert` | `patch` |

Commit types not listed in `rules` produce no release.

## `version_ceiling`

Prevents semrel from releasing at or above this version. Useful for keeping a project below `1.0.0` until you deliberately promote it to stable.

```yaml
version_ceiling: "1.0.0"
```

| Value | Effect |
| --- | --- |
| Not set | No ceiling — semrel bumps freely |
| `"1.0.0"` | Releases are never `>= 1.0.0` |

### `ceiling_strategy`

Controls what happens when the calculated next version would reach or exceed `version_ceiling`.

```yaml
version_ceiling: "1.0.0"
ceiling_strategy: clamp   # clamp (default) | skip | error
```

| Strategy | Behaviour |
| --- | --- |
| `clamp` | Downgrades the bump (major → minor → patch) until the version stays below the ceiling. Logs a warning. |
| `skip` | Does nothing — prints a notice and exits without releasing. |
| `error` | Exits with a non-zero code so the pipeline fails explicitly. |

**Example:** current `0.9.5`, breaking change commit, ceiling `1.0.0`

- `clamp` → releases `0.9.6` (downgraded to patch)
- `skip` → no release, exit 0
- `error` → no release, exit 1

To promote past the ceiling, remove or raise `version_ceiling` in `.semrel.yaml` and commit.

## `plugins`

An ordered list of plugins to load for the release pipeline.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `uses` | string | yes* | Plugin type or install name. semrel resolves to `semrel-plugin-<uses>` |
| `name` | string | no | Optional instance identifier for this plugin entry |
| `path` | string | no | Explicit path to a plugin binary (overrides `uses` resolution) |
| `args` | map | no | Arbitrary key-value pairs passed to the plugin as `SEMREL_PLUGIN_*` environment variables |

*Either `uses` or `path` must be set.

### Plugin resolution

If `path:` is omitted, semrel resolves plugins in this order:

1. `~/.semrel/plugins/semrel-plugin-<uses>`
2. `semrel-plugin-<uses>` from `$PATH`

Use `path:` when you want to pin a specific local binary instead of the default discovery flow.

<Aside type="tip">
  Install official plugins with `semrel plugin install <name>`. By default, the CLI downloads them to `~/.semrel/plugins/`.
</Aside>

### Plugin arguments as environment variables

Every key in `args:` is exposed to the plugin process as an environment variable prefixed with `SEMREL_PLUGIN_`.

```yaml
plugins:
  - uses: github
    args:
      token: ${{ env.GITHUB_TOKEN }}
      owner: MyOrg
      repo: my-repo
```

The plugin receives:

| Config key | Environment variable |
| --- | --- |
| `token` | `SEMREL_PLUGIN_TOKEN` |
| `owner` | `SEMREL_PLUGIN_OWNER` |
| `repo` | `SEMREL_PLUGIN_REPO` |

Plugins also receive release context through environment variables such as `SEMREL_CURRENT_VERSION`, `SEMREL_NEXT_VERSION`, `SEMREL_TAG_NAME`, `SEMREL_CHANGELOG`, and `SEMREL_DRY_RUN`.

semrel plugins are plain executables launched as subprocesses. They do not use gRPC.
