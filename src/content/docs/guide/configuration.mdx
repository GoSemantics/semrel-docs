All semrel behavior is controlled by a single config file in the root of your repository. You can specify a different path with `--config`.

## Supported formats

semrel auto-discovers the config file in this order:

| File | Format |
| --- | --- |
| `.semrel.yaml` / `.semrel.yml` | YAML |
| `.semrel.toml` | TOML |
| `.semrel.json` | JSON |

YAML uses `camelCase` for some keys (e.g. `tagPrefix`). TOML and JSON always use `snake_case` (e.g. `tag_prefix`).

## Full example

<Tabs>
  <TabItem label="YAML (.semrel.yaml)">

      - uses: github-actions

## `tagPrefix` / `tag_prefix`

The prefix prepended to the version number when creating git tags.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    tagPrefix: v   # creates tags like v1.2.3 (default)
    tagPrefix: ""  # creates tags like 1.2.3
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    tag_prefix = "v"   # creates tags like v1.2.3 (default)
    tag_prefix = ""    # creates tags like 1.2.3
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    { "tag_prefix": "v" }
    ```
  </TabItem>
</Tabs>

| Value | Default | Description |
| --- | --- | --- |
| any string | `v` | Prepended to every tag semrel creates |

## `branches`

A list of branches from which releases can be made. Commits on unlisted branches are ignored.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    branches:
      - name: main
      - name: "*.x"
        maintenance: true
      - name: next
        prerelease: beta
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    [[branches]]
    name = "main"

    [[branches]]
    name = "*.x"
    maintenance = true

    [[branches]]
    name = "next"
    prerelease = "beta"
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    {
      "branches": [
        { "name": "main" },
        { "name": "*.x", "maintenance": true },
        { "name": "next", "prerelease": "beta" }
      ]
    }
    ```
  </TabItem>
</Tabs>

### Branch fields

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `name` | string | — | Branch name or glob pattern (e.g. `main`, `"*.x"`, `release/*`) |
| `prerelease` | string | — | Pre-release identifier appended to the version (e.g. `beta` → `1.0.0-beta.1`) |
| `maintenance` | bool | `false` | Restricts this branch to patch-only bumps. Also auto-detected from `N.x` / `N.M.x` patterns |

<Aside type="tip">
  Maintenance branches like `1.x` or `2.3.x` are auto-detected — you don't need to set `maintenance: true` explicitly for name patterns that match `N.x` or `N.M.x`.
</Aside>

## `rules`

Maps conventional commit types to semver bump levels. If omitted, semrel uses the built-in defaults.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    rules:
      - type: feat
        bump: minor
      - type: fix
        bump: patch
      - type: perf
        bump: patch
      - type: revert
        bump: patch
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    [[rules]]
    type = "feat"
    bump = "minor"

    [[rules]]
    type = "fix"
    bump = "patch"

    [[rules]]
    type = "perf"
    bump = "patch"

    [[rules]]
    type = "revert"
    bump = "patch"
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    {
      "rules": [
        { "type": "feat",   "bump": "minor" },
        { "type": "fix",    "bump": "patch" },
        { "type": "perf",   "bump": "patch" },
        { "type": "revert", "bump": "patch" }
      ]
    }
    ```
  </TabItem>
</Tabs>

### Rule fields

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `type` | string | yes | Conventional commit type (e.g. `feat`, `fix`, `perf`) |
| `bump` | string | yes | One of `major`, `minor`, `patch` |

<Aside type="note">
  Breaking changes (`BREAKING CHANGE` footer or `!` suffix like `feat!:`) always result in a `major` bump regardless of rules.
</Aside>

### Default rules

When `rules:` is absent, semrel uses:

| Type | Bump |
| --- | --- |
| `feat` | `minor` |
| `fix` | `patch` |
| `perf` | `patch` |
| `revert` | `patch` |

Commit types not listed in `rules` produce no release.

## `commit_changelog`

Controls whether semrel commits `CHANGELOG.md` back to the repository before creating the release tag.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    commit_changelog: true   # default
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    commit_changelog = true
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    { "commit_changelog": true }
    ```
  </TabItem>
</Tabs>

| Value | Behaviour |
| --- | --- |
| `true` (default) | semrel writes CHANGELOG.md, commits it as `chore(changelog): update for vX.Y.Z [skip ci]`, then creates the tag on that commit |
| `false` | semrel writes CHANGELOG.md locally but does not commit it (useful if you manage the changelog externally) |

<Aside type="tip">
  The `[skip ci]` suffix in the changelog commit message prevents most CI systems from triggering a new pipeline run for the changelog commit itself.
</Aside>

## `tag_exists_strategy`

Controls what semrel does when the computed next version tag already exists locally.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    tag_exists_strategy: update-changelog   # default
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    tag_exists_strategy = "update-changelog"
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    { "tag_exists_strategy": "update-changelog" }
    ```
  </TabItem>
</Tabs>

| Strategy | Behaviour |
| --- | --- |
| `update-changelog` (default) | Updates and commits `CHANGELOG.md` for the existing version, then exits without error. Idempotent — useful for repair runs. |
| `skip` | Exits silently without any changes. |
| `error` | Exits with a non-zero code. Use this if you want strict one-shot pipelines. |

**Example scenario:** you manually created `v0.1.0` to bootstrap the project. The next pipeline run detects the tag exists and, with the default strategy, regenerates and commits the changelog for that version — no duplicate release is created.

## `version_ceiling`

Prevents semrel from releasing at or above this version. Useful for keeping a project below `1.0.0` until you deliberately promote it to stable.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    version_ceiling: "1.0.0"
    ceiling_strategy: clamp
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    version_ceiling = "1.0.0"
    ceiling_strategy = "clamp"
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    {
      "version_ceiling": "1.0.0",
      "ceiling_strategy": "clamp"
    }
    ```
  </TabItem>
</Tabs>

| Value | Effect |
| --- | --- |
| Not set | No ceiling — semrel bumps freely |
| `"1.0.0"` | Releases are never `>= 1.0.0` |

### `ceiling_strategy`

Controls what happens when the calculated next version would reach or exceed `version_ceiling`.

| Strategy | Behaviour |
| --- | --- |
| `clamp` | Downgrades the bump (major → minor → patch) until the version stays below the ceiling. Logs a warning. |
| `skip` | Does nothing — prints a notice and exits without releasing. |
| `error` | Exits with a non-zero code so the pipeline fails explicitly. |

**Example:** current `0.9.5`, breaking change commit, ceiling `1.0.0`

- `clamp` → releases `0.9.6` (downgraded to patch)
- `skip` → no release, exit 0
- `error` → no release, exit 1

To promote past the ceiling, remove or raise `version_ceiling` in your config file and commit.

## `plugins`

An ordered list of plugins to load for the release pipeline.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `uses` | string | yes* | Plugin type or install name. semrel resolves to `semrel-plugin-<uses>` |
| `name` | string | no | Optional instance identifier (useful when the same plugin appears multiple times) |
| `path` | string | no | Explicit path to a plugin binary (overrides `uses` resolution) |
| `args` | map | no | Arbitrary key-value pairs passed to the plugin as `SEMREL_PLUGIN_*` environment variables |
| `phase` | string | no | When to run: `condition`, `pre-tag`, or `release` (default) |

*Either `uses` or `path` must be set.

### Plugin phases

| Phase | When it runs | Typical use |
| --- | --- | --- |
| `condition` | After config load, **before** any commit analysis or tag creation | `github-actions`, `gitlab-ci` — abort if environment is wrong |
| `pre-tag` | After version calculation and changelog write, **before** the git tag is created | `gobinary` — update version file and commit so the tagged source contains the real version |
| `release` | After the tag is created (default) | `github`, `hook-slack`, `updater-npm` — publish and notify |

<Tabs>
  <TabItem label="YAML">
    ```yaml
    plugins:
      # Gate: abort if not running in GitHub Actions
      - uses: github-actions
        phase: condition

      # Pre-tag: update Go version variable before tagging
      - uses: gobinary
        phase: pre-tag
        args:
          file: internal/version/version.go
          var_name: Version

      # Release: create GitHub release with changelog
      - uses: github
        phase: release   # default, can be omitted
        args:
          token: ${{ env.GITHUB_TOKEN }}
          owner: MyOrg
          repo: my-repo

      # Hook: notify Slack after release
      - uses: slack
        args:
          webhook_url: ${{ env.SLACK_WEBHOOK }}
    ```
  </TabItem>
  <TabItem label="TOML">

### Plugin resolution

If `path:` is omitted, semrel resolves plugins in this order:

1. `~/.semrel/plugins/semrel-plugin-<uses>`
2. `semrel-plugin-<uses>` from `$PATH`

Use `path:` when you want to pin a specific local binary instead of the default discovery flow.

<Aside type="tip">
  Install official plugins with `semrel plugin install <name>`. By default, the CLI downloads them to `~/.semrel/plugins/`.
</Aside>

### Plugin arguments as environment variables

Every key in `args:` is exposed to the plugin process as an environment variable prefixed with `SEMREL_PLUGIN_`.

<Tabs>
  <TabItem label="YAML">
    ```yaml
    plugins:
      - uses: github
        args:
          token: ${{ env.GITHUB_TOKEN }}
          owner: MyOrg
          repo: my-repo
    ```
  </TabItem>
  <TabItem label="TOML">
    ```toml
    [[plugins]]
    uses = "github"
    [plugins.args]
    token = "${GITHUB_TOKEN}"
    owner = "MyOrg"
    repo = "my-repo"
    ```
  </TabItem>
  <TabItem label="JSON">
    ```json
    {
      "plugins": [{
        "uses": "github",
        "args": {
          "token": "${GITHUB_TOKEN}",
          "owner": "MyOrg",
          "repo": "my-repo"
        }
      }]
    }
    ```
  </TabItem>
</Tabs>

The plugin receives:

| Config key | Environment variable |
| --- | --- |
| `token` | `SEMREL_PLUGIN_TOKEN` |
| `owner` | `SEMREL_PLUGIN_OWNER` |
| `repo` | `SEMREL_PLUGIN_REPO` |

Plugins also receive release context through standard environment variables:

| Variable | Description |
| --- | --- |
| `SEMREL_CURRENT_VERSION` | The version before this release |
| `SEMREL_NEXT_VERSION` | The new version being released |
| `SEMREL_TAG_NAME` | The full tag name including prefix (e.g. `v1.2.3`) |
| `SEMREL_CHANGELOG` | Path to the written `CHANGELOG.md` file |
| `SEMREL_DRY_RUN` | `true` if semrel was invoked with `--dry-run` |

semrel plugins are plain executables launched as subprocesses. They do not use gRPC.
